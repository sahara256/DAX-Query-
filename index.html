What is DAX?

DAX (Data Analysis Expressions) is the formula language used in Power BI, Power Pivot, and Analysis Services. It lets you create custom calculations, aggregations, and business logic on top of your data model.

Two types of DAX calculations

Measures — Dynamic calculations evaluated in the context of filters (e.g., Total Sales). They don't add a column to the table.

Calculated Columns — Row-by-row calculations stored in the table (e.g., Profit = Sales - Cost).

DAX is case-insensitive. SUM = sum = Sum. But use UPPERCASE for functions by convention.

DAX Syntax Rules

-- Table references use square brackets for columns
Sales[Revenue]      -- Fully qualified: Table[Column]
[Revenue]         -- Short form (inside same table context)

-- Functions follow this pattern:
FUNCTION_NAME( argument1, argument2 )

-- Measures are defined with :=
Total Sales := SUM( Sales[Revenue] )

Basic Functions

SUM, AVERAGE, MIN, MAX, COUNT

-- Sum all values in a column
Total Revenue := SUM( Sales[Revenue] )

-- Average of a column
Avg Sale := AVERAGE( Sales[Revenue] )

-- Minimum and Maximum
Min Price := MIN( Products[Price] )
Max Price := MAX( Products[Price] )

-- Count rows (non-blank)
Transaction Count := COUNT( Sales[OrderID] )

-- Count all rows including blanks
Row Count := COUNTA( Sales[OrderID] )

-- Count distinct values
Unique Customers := DISTINCTCOUNT( Sales[CustomerID] )

Basic Arithmetic

-- Calculated Column: Profit
Profit = Sales[Revenue] - Sales[Cost]

-- Calculated Column: Profit Margin %
Profit Margin = DIVIDE( Sales[Profit], Sales[Revenue], 0 )

-- DIVIDE is safer than "/" — handles division by zero
-- Third argument = value to return when dividing by 0
Always use DIVIDE(numerator, denominator, 0) instead of the / operator to avoid divide-by-zero errors.

Text & Date Functions

Text Functions

-- Concatenate text
Full Name = CONCATENATE( Customers[FirstName], Customers[LastName] )
-- Or use &
Full Name = Customers[FirstName] & " " & Customers[LastName]

-- Upper / Lower case
Upper Name = UPPER( Customers[Name] )

-- Extract characters
First3 = LEFT( Products[SKU], 3 )
Last3  = RIGHT( Products[SKU], 3 )

-- Find and Replace
Clean = SUBSTITUTE( Customers[Phone], "-", "" )

-- String length
Name Length = LEN( Customers[Name] )

Date Functions

-- Extract parts of a date
Year  = YEAR( Sales[OrderDate] )
Month = MONTH( Sales[OrderDate] )
Day   = DAY( Sales[OrderDate] )

-- Today's date
Today = TODAY()
Now   = NOW()

-- Days between dates
Days Since Order = DATEDIFF( Sales[OrderDate], TODAY(), DAY )

-- Format date as text
Month Year = FORMAT( Sales[OrderDate], "MMM YYYY" )

Logical Functions

IF

-- Basic IF
Profit Flag = IF( Sales[Profit] > 0, "Profitable", "Loss" )

-- Nested IF
Rating = IF( Sales[Score] >= 90, "A",
          IF( Sales[Score] >= 75, "B",
          IF( Sales[Score] >= 60, "C", "F" )))

SWITCH

-- SWITCH is cleaner than nested IF
Quarter Label =
SWITCH(
    MONTH( Sales[OrderDate] ),
    1, "Q1", 2, "Q1", 3, "Q1",
    4, "Q2", 5, "Q2", 6, "Q2",
    7, "Q3", 8, "Q3", 9, "Q3",
    "Q4"  -- Default value
)

-- SWITCH(TRUE()) replaces complex nested IFs
Performance =
SWITCH( TRUE(),
    Sales[Revenue] > 10000, "Excellent",
    Sales[Revenue] > 5000,  "Good",
    Sales[Revenue] > 1000,  "Average",
    "Poor"
)

AND, OR, NOT

-- Logical operators
Premium Customer =
IF( AND( Customers[Orders] > 10, Customers[TotalSpend] > 5000 ),
    "Premium", "Standard" )

-- Using && and || operators (same as AND/OR)
Eligible = IF( Customers[Age] >= 18 && Customers[Active] = TRUE(), "Yes", "No" )

FILTER & CALCULATE

These are the most powerful functions in DAX. CALCULATE modifies the filter context of a calculation.

CALCULATE

-- Syntax: CALCULATE(expression, filter1, filter2, ...)

-- Total sales for only "Electronics" category
Electronics Sales :=
CALCULATE(
    SUM( Sales[Revenue] ),
    Products[Category] = "Electronics"
)

-- Multiple filters (AND logic)
East Electronics :=
CALCULATE(
    SUM( Sales[Revenue] ),
    Products[Category] = "Electronics",
    Regions[Area] = "East"
)

FILTER

-- FILTER returns a table of rows that match a condition
-- Used inside CALCULATE for complex conditions

High Value Sales :=
CALCULATE(
    SUM( Sales[Revenue] ),
    FILTER( Sales, Sales[Revenue] > 1000 )
)

-- FILTER with multiple conditions
Premium Orders :=
CALCULATE(
    COUNTROWS( Sales ),
    FILTER(
        Sales,
        Sales[Revenue] > 500 && Sales[Quantity] > 2
    )
)

ALL & ALLEXCEPT

-- ALL removes all filters from a table/column
% of Total Sales :=
DIVIDE(
    SUM( Sales[Revenue] ),
    CALCULATE( SUM( Sales[Revenue] ), ALL( Sales ) )
)

-- ALLEXCEPT keeps filters on specified columns
% of Category :=
DIVIDE(
    SUM( Sales[Revenue] ),
    CALCULATE( SUM( Sales[Revenue] ), ALLEXCEPT( Products, Products[Category] ) )
)
Think of CALCULATE as "evaluate this expression but first apply these filter changes to the model."

Time Intelligence

Time intelligence functions require a proper Date table marked as a Date Table in Power BI with a continuous date column.

Year-to-Date (YTD), Quarter-to-Date (QTD), Month-to-Date (MTD)

-- Year to Date Sales
Sales YTD :=
CALCULATE(
    SUM( Sales[Revenue] ),
    DATESYTD( Calendar[Date] )
)

-- Quarter to Date
Sales QTD :=
CALCULATE( SUM( Sales[Revenue] ), DATESQTD( Calendar[Date] ) )

-- Month to Date
Sales MTD :=
CALCULATE( SUM( Sales[Revenue] ), DATESMTD( Calendar[Date] ) )

-- YTD with custom fiscal year ending June 30
Sales Fiscal YTD :=
CALCULATE(
    SUM( Sales[Revenue] ),
    DATESYTD( Calendar[Date], "06-30" )
)

Same Period Last Year (SPLY)

-- Sales in the same period last year
Sales LY :=
CALCULATE(
    SUM( Sales[Revenue] ),
    SAMEPERIODLASTYEAR( Calendar[Date] )
)

-- Year-over-Year Growth %
YoY Growth % :=
DIVIDE(
    [Total Sales] - [Sales LY],
    [Sales LY],
    0
)

DATEADD — Flexible Period Shifting

-- Sales 1 month ago
Sales Last Month :=
CALCULATE(
    SUM( Sales[Revenue] ),
    DATEADD( Calendar[Date], -1, MONTH )
)

-- Sales 3 months ago
Sales 3M Ago :=
CALCULATE(
    SUM( Sales[Revenue] ),
    DATEADD( Calendar[Date], -3, MONTH )
)

Rolling 12-Month Total

Rolling 12M Sales :=
CALCULATE(
    SUM( Sales[Revenue] ),
    DATESINPERIOD(
        Calendar[Date],
        LASTDATE( Calendar[Date] ),
        -12,
        MONTH
    )
)

Iterator Functions (X Functions)

Iterator functions loop over each row of a table, evaluate an expression for each row, and then aggregate the result. They are identified by the "X" suffix: SUMX, AVERAGEX, MAXX, MINX, COUNTX.

SUMX

-- SUMX: calculate per-row then sum
-- Total Revenue = Quantity * Unit Price (per row)
Total Revenue :=
SUMX(
    Sales,                                       -- Table to iterate
    Sales[Quantity] * Sales[UnitPrice]           -- Expression per row
)

-- Compare to: this would be WRONG because columns can't multiply directly
-- Wrong: SUM(Sales[Quantity]) * SUM(Sales[UnitPrice])  ← not the same!

AVERAGEX

-- Average order value (per transaction)
Avg Order Value :=
AVERAGEX(
    Sales,
    Sales[Quantity] * Sales[UnitPrice]
)

RANKX

-- Rank products by total sales (1 = highest)
Sales Rank :=
RANKX(
    ALL( Products[ProductName] ),    -- Table to rank against
    [Total Sales],                    -- Expression to rank by
    ,                                 -- Value (blank = use expression)
    DESC,                             -- Order (DESC = highest = rank 1)
    Dense                             -- Tie-breaking: Dense skips no ranks
)

Row Context vs Filter Context

🔵 Filter Context

The set of active filters when a measure is evaluated — determined by slicers, report filters, rows/columns in a matrix visual. Measures always run in filter context.

🟡 Row Context

Exists when iterating over a table (like SUMX or in a Calculated Column). It knows "which row I'm currently on."

RELATED — Cross-table lookups in row context

-- In a Calculated Column on Sales table,
-- get the Category from related Products table
Category = RELATED( Products[Category] )

-- RELATED follows the "many" side to the "one" side
-- Works inside row context (calculated columns or SUMX iterators)

RELATEDTABLE — One to Many

-- From Products table (one), count related Sales rows (many)
Order Count = COUNTROWS( RELATEDTABLE( Sales ) )

EARLIER — Nested row context

-- Rank within each group using EARLIER
Rank in Category =
COUNTROWS(
    FILTER(
        Products,
        Products[Category] = EARLIER( Products[Category] ) &&
        Products[Sales] >= EARLIER( Products[Sales] )
    )
)
-- EARLIER() refers to the outer row context

Advanced

Advanced Functions

CALCULATETABLE

-- Returns a filtered TABLE (not a scalar value)
Top Products Table =
CALCULATETABLE(
    Products,
    Products[Category] = "Electronics",
    Products[Price] > 500
)

TOPN

-- Get the top 5 products by sales
Top 5 Revenue :=
CALCULATE(
    SUM( Sales[Revenue] ),
    TOPN( 5, ALL( Products ), [Total Sales] )
)

SUMMARIZE & ADDCOLUMNS

-- SUMMARIZE groups data like SQL GROUP BY
Category Summary =
SUMMARIZE(
    Sales,
    Products[Category],
    "Total Revenue", SUM( Sales[Revenue] )
)

-- ADDCOLUMNS adds computed columns to a table
Enriched =
ADDCOLUMNS(
    SUMMARIZE( Sales, Products[Category] ),
    "Category Sales", CALCULATE( SUM( Sales[Revenue] ) )
)

USERELATIONSHIP — Activating Inactive Relationships

-- When you have multiple date columns (OrderDate vs ShipDate)
-- and only one relationship can be active at a time

Sales by Ship Date :=
CALCULATE(
    SUM( Sales[Revenue] ),
    USERELATIONSHIP( Sales[ShipDate], Calendar[Date] )
)

SELECTEDVALUE & HASONEVALUE

-- Returns the single selected value (or alternate if multiple)
Selected Year :=
SELECTEDVALUE( Calendar[Year], "All Years" )

-- Use HASONEVALUE to check if a single value is filtered
Dynamic Title :=
IF(
    HASONEVALUE( Products[Category] ),
    "Sales for: " & VALUES( Products[Category] ),
    "Total Sales - All Categories"
)

CROSSFILTER

-- Change cross-filter direction dynamically
Customers with Purchases :=
CALCULATE(
    DISTINCTCOUNT( Sales[CustomerID] ),
    CROSSFILTER( Sales[CustomerID], Customers[CustomerID], BOTH )
)

Variables (VAR / RETURN)

Variables make DAX formulas readable, easier to debug, and more performant (a variable is evaluated only once).

-- Always use VAR...RETURN for readable, efficient DAX
YoY Growth % :=
VAR CurrentSales = SUM( Sales[Revenue] )
VAR LastYearSales =
    CALCULATE(
        SUM( Sales[Revenue] ),
        SAMEPERIODLASTYEAR( Calendar[Date] )
    )
VAR Growth = DIVIDE( CurrentSales - LastYearSales, LastYearSales, 0 )
RETURN
    IF( ISBLANK( LastYearSales ), BLANK(), Growth )

-- Variables can hold tables too!
Top Customers Revenue :=
VAR TopCustomers =
    TOPN( 10, ALL( Customers ), [Total Sales] )
RETURN
    CALCULATE( SUM( Sales[Revenue] ), TopCustomers )

Best Practices

DAX Best Practices

PracticeWhy
Use DIVIDE() instead of /Handles divide-by-zero gracefully
Use VAR / RETURNImproves readability and performance
Prefer Measures over Calculated ColumnsMeasures are dynamic, don't consume memory
Avoid FILTER(ALL(...)) for simple filtersUse direct predicates in CALCULATE instead
Always have a Date TableRequired for Time Intelligence functions
Use ISBLANK() not = BLANK()More reliable blank detection
Avoid EARLIER when possibleUse SUMX + FILTER pattern instead
Avoid bi-directional relationships widelyCan cause ambiguity; use CROSSFILTER selectively
